### 4. Initial Build Timing is Fragile (TIMING BUG)

**Severity**: üü† HIGH (occasionally manifests)

**Current Code** (dev.js lines 35-39):
```javascript
// Initial build
runBuild(() => {
    bs.reload(); // Reload called immediately after build completes
});
```

**The Race Condition**:
1. BrowserSync starts listening for connections
2. Initial build starts via `runBuild()` in background
3. Build completes, callback fires: `bs.reload()`
4. But browser client may not be connected yet
5. Reload signal queues but may be missed

**In Practice**: 
- On fresh `npm run dev`, first page load might not trigger initial reload
- User must manually refresh browser
- Happens intermittently, hard to reproduce
- Only issue if you're developing actively during startup

**Fix Required**: Ensure build completes before BrowserSync accepts connections, or queue reloads properly

## Architecture Problems

### 5. Markdown ‚Üí HTML Generation is Lossy & Unmaintainable

**Severity**: üü† HIGH

**Current Flow**:
```
DORA_AI_Paradox.md
    ‚Üì (markdown-it default config)
DORA_AI_Paradox.html (with inline styles)
    ‚Üì (read and embedded)
docs/index.html (monolithic single-page app)
```

**The Problems**:

1. **Inline Styles Violate Your Own Linter**
   - Markdown-it renders tables with `style="text-align:left"` by default
   - Your HTML linter forbids inline styles
   - Generated HTML fails validation
   - You ignored this, created technical debt

2. **Cannot Separate Content from Presentation**
   - To change table styling, must regenerate HTML from markdown
   - Cannot hand-edit generated HTML (breaks build contract)
   - Couples content updates to styling decisions
   - Example: Want to add a `<thead>` class? Must edit markdown and regenerate

3. **No Markdown-it Configuration**
   ```javascript
   // build.js line 5
   const md = new MarkdownIt();  // Zero config!
   ```
   - Using default settings for everything
   - No custom renderers
   - No table styling overrides
   - No LaTeX support (report uses `$\downarrow -7.2\%$` syntax)

**Better Approach**:
```javascript
const md = new MarkdownIt();
md.renderer.rules.table_open = () => '<table class="table">';
md.renderer.rules.th_open = () => '<th class="table__cell table__cell--header">';
// Now generated HTML uses BEM classes, passes linter
```

### 6. Single-Page App Architecture is Wrong for Static Content

**Severity**: üü° MEDIUM

**Current Design**:
- 7 markdown files converted to HTML
- All embedded into single `index.html`
- JavaScript handles tab switching via hash fragments
- All content loads on first request

**Problems**:

1. **Page Size**: All 7 documents load at once (unnecessary)
   - Meeting 0: 44 lines
   - Meeting 1: 68 lines
   - Visual Summary: 218 lines
   - Plus facilitator guide, overview, etc.
   - Total: 637+ lines of content loaded regardless of which tab user visits

2. **SEO is Degraded**:
   - All content at single URL (`/`)
   - Search engines see only first tab content
   - Cannot deep-link to specific meeting (`/meeting1` doesn't work)
   - Social sharing doesn't preview specific meeting

3. **Accessibility Issues**:
   - Screen readers must parse massive 7-in-1 DOM
   - Tab navigation requires JavaScript
   - No semantic separation between meetings
   - Tab system uses data attributes, not semantic HTML

4. **Scaling Problem**:
   - Adding Meeting 4, 5, 6 requires:
     - Create meeting4.md
     - Edit build.js to add to filesToConvert array
     - Edit dev.js to update port? (No, but BrowserSync config)
     - Recreate index.html manually
   - This doesn't scale; there's no auto-discovery

**Better Approach**: 
- Static multi-page site: `meeting0.html`, `meeting1.html`, etc. as root files
- Or: Use a proper static site generator with templating (Hugo, Jekyll)
- Or: Keep current but generate individual pages + index with proper links

## Closing Statement

**This project is a working prototype, not a production system.** It demonstrates an interesting idea (markdown-driven book club) but lacks the fundamentals for collaboration: working git history, error visibility, testing, documentation, and automation.

**The two critical blockers are:**
1. **Git repository doesn't contain source files** ‚Üí Can't clone & run
2. **Build errors are silent** ‚Üí Developers can't see what's wrong

Fix these two, and the project becomes usable. Then address the process gaps (testing, CI/CD, documentation) to make it maintainable.
